# 线上整改、优化、重构 —— 完整补充规范（最终版）

1. **需求必须集中收集、统一排期、分批上线**

   不零散、不临时、不插队，避免频繁发布带来风险。

   

2. **严禁见一个改一个、临时插需求、即兴修改**

   防止逻辑碎片化、回归不充分、线上不稳定。

   

3. **大重构 / 大优化必须小步快跑、增量上线**

   拆小任务、逐个验证、不做大包发布。

   

4. **必须写需求文档 + 设计方案 + 验收标准**

   无方案不开发，无验收不上线。

   

5. **任何重构 / 优化必须保证原有行为完全不变**

   功能、入参、出参、异常、性能阈值都不能变。

   

6. **必须做回归测试 + 核心链路自动化测试**

   重构最容易踩老坑，必须全量回归。

   

7. **重构必须保留原有日志、监控、告警点位**

   不能因为重构导致可观测性下降。

   

8. **大改动必须灰度 / 放量 / 观察**

   先 1%→10%→50%→100%，出现问题立即回滚。

   

9. **必须有回滚方案，回滚时间 <5 分钟**

   重构越复杂，回滚机制越重要。

   

10. **性能优化必须有基线数据，优化后必须对比验证**

    无数据不叫优化，叫瞎改。

    

11. **代码整改必须遵循统一规范，不能越整越乱**

    风格、结构、分层、接口必须保持一致。

    

12. **禁止在重构中夹带新功能**

    重构 = 结构优化；新需求 = 单独批次。

    混包必出线上事故。

    

13. **重构必须同步更新文档、接口定义、架构图**

    代码与文档必须一致，避免后续人员误解。

    

14. **核心链路重构必须做线上压测 / 流量回放**

    确保性能不下降、不引入新瓶颈。

    

15. **所有改动必须关联工单、可追溯**

    谁改、改什么、为什么改、何时上、回滚方式，全部留痕。

# 线上优化 & 重构 标准管理流程（5 步闭环・可直接作为团队制度）

## 一、需求收集与合并（杜绝见一个改一个）

1. 统一入口收集问题：BUG、性能、安全、技术债、体验优化
2. 按**影响面、风险、成本**分级：P0/P1/P2/P3
3. 同类需求合并，**分批打包**，不零散上线
4. 禁止临时插队、即兴修改

## 二、方案设计与文档（无方案不开发）

1. 明确：目标、范围、改动点、不变点

2. 输出：

   - 需求说明
   - 架构 / 接口 / 流程设计
   - 风险点与防护措施
   - **回滚方案**（必须有）

   

3. 大重构必须：**新旧兼容、增量替换、不破坏原有行为**

## 三、分批实施 & 小步快跑（禁止大包上线）

1. 大重构拆成**最小可验证单元**
2. 优先：工具 / 通用层 → 业务逻辑 → 接口层
3. 每步只做一件事：**重构 ≠ 新功能**，严禁混包
4. 每步上线后观察指标，再继续下一步

## 四、验收标准（无验收不上线）

1. 功能验收：原有行为**完全不变**
2. 性能验收：有基线、有对比、有提升
3. 异常验收：日志 / 监控 / 告警正常
4. 安全验收：无注入、无越权、无泄露
5. 兼容验收：上下游无影响

## 五、上线管控与观测（确保稳定）

1. 灰度 / 放量：1% → 10% → 50% → 100%
2. 核心观测：错误率、响应时间、流量、日志
3. 异常**立即回滚**，回滚时间 <5 分钟
4. 上线完成后同步更新：文档、架构图、接口规范
