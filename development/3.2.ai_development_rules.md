## 1.代码结构与注释规范

### 1.1脚本头部总述

- 必须在脚本首行，用**单行 / 多行注释**清晰说明**所有核心接口（类、函数）的整体职责**，一句话概括脚本核心功能与适用场景。

### **1.2类与函数注释**

- **类**：必须写 `docstring`，明确类的**核心职责、依赖关系、使用场景**，不超过 3 句话。
- **函数**：必须写 `docstring`，包含**功能描述、输入参数（含类型、含义、必填 / 可选）、返回值（含类型、含义）、异常说明**；必须补充**调用示例**（1-2 个典型场景）。

### 1.3核心代码注释

- 关键逻辑行必须加注释，覆盖：**边界条件处理、异常分支、性能瓶颈优化点、关键计算逻辑**。
- 注释简洁准确，不冗余，不重复代码逻辑。



## 2.接口开发规范

### 2.1核心思想

- **公共接口模块 = 系统唯一对外服务出口**，统一协议、统一开放、统一维护。
- **多接口聚合 / 组装只在公共接口层完成**，下游绝不参与组装。
- **公共层与下游完全解耦**：公共层不侵入内部，下游不感知公共层。
- 公共接口追求：**通用、稳定、安全、可复用、可兼容**。

### 2.2代码组织规则

- **按业务域 / 功能模块对函数分组**，确保功能内聚、边界清晰。
- **每组功能封装为类**，一个类只负责一类能力。
- **函数保持短小精简**，严格遵循**单一职责原则**。
- **禁止深层嵌套逻辑**，复杂逻辑拆分为多个细粒度方法。
- **统一命名规范**：类、方法、变量语义清晰、风格一致。

### 2.3 接口开放与调用规则

- 类中方法**统一使用 @classmethod**，保持对外接口风格一致。
- **公共接口严禁直接调用下游私有函数、变量、内部结构、数据**。
- 所有依赖下游的操作，**必须由下游模块自行实现并提供标准入口**。
- **多接口编排、聚合、数据组装逻辑，只允许在公共接口模块实现**，下游只提供原子能力。
- 公共接口保持通用、稳定、与具体业务解耦，**可被全系统任意模块安全复用**。
- 公共接口只暴露标准服务能力，**不绑定任何下游特定实现**。

### 2.4 架构约束（铁律）

- **外部 → 公共接口 → 下游内部**，唯一调用链路。

- 下游内部**不向上依赖、不反向调用、不输出内部结构**。

- 公共接口**只做编排、校验、转换、聚合、异常标准化**。

- 任何变更必须**保持兼容**，不改变原有入参、出参、行为逻辑。

  

## 3.异常与日志规范

### **3.1异常处理**

- 必须覆盖**输入校验、接口调用、资源操作（如数据库、文件）、数据解析**等核心环节的异常。
- 统一使用 `try-except` 捕获特定异常（禁止裸 except），异常后需**记录日志**并给出**友好提示**。
- 异常处理时不要返回 null，直接抛出相应错误，便于排查问题。

### 3.2**日志规范**

- 必须引入日志模块（如 `logging`），配置**日志级别、格式（含时间、模块、行号、异常信息）、输出目标（控制台 / 文件）**。
- 关键节点（接口调用、资源初始化 / 销毁、异常发生）必须打日志，日志信息清晰可追溯。



## 4.测试规范

### 4.1**单元测试**

- 针对**核心函数、类方法**编写单元测试，覆盖：**正常场景、边界场景、异常场景**。
- 使用 Python 内置 `pytest` 框架，测试用例命名清晰，每个测试函数对应一个验证点。

### 4.2**集成测试**

- 针对**完整接口链路、模块交互**（如调用数据库、文件上传下载全流程）编写集成测试。
- 模拟真实运行环境，验证**接口连通性、数据一致性、异常容错性**。



## 5.补充最佳实践

### 5.1**类型提示**

- 全脚本强制使用 Python 类型提示（`int/str/list` 等），明确参数、返回值类型，提升代码可读性与可维护性。

### 5.2**依赖管理**

- 脚本依赖的第三方库需在脚本头部注明，或配套提供 `requirements.txt`，明确版本要求。

### 5.3**可配置性**

- 核心配置（如数据库地址、端口、超时时间）**硬编码禁止**，需通过配置文件 / 环境变量管理，支持灵活部署。

### 5.4**文档与可调试性**

- 脚本配套生成接口文档（如 FastAPI 自动生成 `/docs`），方便调用方理解与调试。
- 代码结构清晰，函数 / 类职责单一，避免 “大而全” 的复杂模块。
